<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| ParseResult.messages → DOM | Message content (text, sender, timestamp) flows from parsed strings to DOM elements |
| ParseResult.plainText → Clipboard / File | Plain text string leaves browser context via clipboard API or Blob download |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | XSS via message content in renderMessage | mitigate | `renderMessage` uses `textContent` exclusively for all three spans (timestamp, sender, body); this plan adds no innerHTML paths; verified by automated grep |
| T-03-02 | Information Disclosure | plainText clipboard content leaves device | accept | This is intentional product behavior — user copies to paste into Claude; no PII beyond what the user chose to export; local-only tool; no network involved |
| T-03-03 | Denial of Service | Very large chat log causing DOM hang on render | accept | Realistic WhatsApp exports have at most tens of thousands of messages; DOM append loop is O(n) and fast enough; Phase 2 can virtualise if needed |
| T-03-04 | Tampering | URL object URL not revoked (memory leak) | mitigate | `URL.revokeObjectURL(url)` called immediately after `a.click()` in downloadTxt; verified by automated grep |
| T-03-05 | Denial of Service | Copy button spam creating many clipboard promises | mitigate | Copy handler is async; repeated clicks race but each call is idempotent (overwrites clipboard with same content); no state corruption possible |
| T-03-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages installed in this plan; all additions are CSS rules and JS extending existing files |
</threat_model>
