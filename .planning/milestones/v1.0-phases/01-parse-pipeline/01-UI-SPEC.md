## Screens and States

Phase 1 defines exactly five UI states. Each is a full-screen replacement — no modals, no overlays.

### State 1 — Upload Screen (landing state)

The default state on launch. One primary action.

**Layout:**
- Centered single-column, max-width 640px, horizontally and vertically centered in the viewport
- App title at top: "VoiceFill" — Display size, normal weight
- Tagline below title: "Reconstruct your WhatsApp voice messages" — Body size, muted (secondary text color)
- Drag-and-drop zone: full-width within the column, min-height 160px, dashed border in accent color at 50% opacity, rounded 4px corners
- Drop zone label text (two lines, centered inside zone):
  - Line 1: "Drop your WhatsApp export ZIP here" — Label size, normal weight
  - Line 2: "or" — Body size, muted
- File picker button below "or": "Browse for file" — Label size, normal weight, accent color border, transparent background
- Fine print below button: "Accepts .zip from WhatsApp 'Export Chat' → 'Attach Media'" — Body size, muted

**Interaction:**
- Drag-enter: drop zone background shifts to accent color at 10% opacity; border opacity increases to 100%
- Drag-leave: returns to default state immediately
- Drop or file picker selection: transitions to Processing Screen
- Accepted file types: `.zip` only; silently ignore non-ZIP drops (no error — drop zone returns to idle)

### State 2 — Processing Screen

Brief, calm. Phase 1 parsing is fast (no ML). Displayed only while ZIP is being read and parsed.

**Layout:**
- Same centered column as Upload Screen
- Single status line, vertically centered in viewport: "Reading export" followed by an animated ellipsis
- Animated ellipsis: cycles through `·`, `··`, `···` at 600ms per frame — CSS animation, no JS timer

**Interaction:**
- No user interaction available on this screen
- Transition to Results Screen or "Without Media" Error Screen on completion
- If parsing completes faster than 300ms, show Processing Screen for a minimum of 300ms to prevent a jarring flash

### State 3 — Results Screen

The primary output state. Two sub-zones: summary header and scrollable chat log.

**Layout:**
- Full-width, scrollable page (not fixed height)
- Sticky header bar (top of viewport, full width):
  - Left: "VoiceFill" — Label size
  - Right: action row — "Copy to clipboard" button + "Download .txt" button
  - Header background: dominant color with a 1px bottom border in secondary color
- Below header: summary line — "X of Y voice messages identified" — Body size, normal weight, 32px top margin
- Below summary: reconstructed chat log block — monospace, scrollable, full-width within 800px max-width column

**Chat Log Rendering Rules:**
- Each line rendered exactly as parsed from `_chat.txt`, preserving the original WhatsApp format
- Timestamps: `[date, time]` prefix — Body size, muted (secondary text color, 60% opacity)
- Sender name: extracted from each line — Label size, accent color (`#8b5e3c`), normal weight
- Message body: Body size, dark ink (`#2c2016`), normal weight
- Voice message annotations — visually distinct from regular message bodies:
  - `[Voice message: transcription pending]` — italic, muted (secondary text color), Body size
  - `[Audio unreadable]` — italic, muted, Body size
  - `[Audio file missing]` — italic, muted, Body size
  - `[Audio not available]` — italic, muted, Body size
- Line spacing within chat log: 1.6 (slightly looser than body — improves scannability of a dense log)
- Paragraph gap between consecutive messages from different senders: 8px additional margin

**Action Buttons (sticky header):**
- "Copy to clipboard": primary style — accent color background (`#8b5e3c`), light text (`#f5f0e8`), 8px vertical padding, 16px horizontal padding, no border-radius (sharp corners reinforce typewriter aesthetic)
- "Download .txt": secondary style — transparent background, accent color border (1px solid), accent color text, same padding
- Button hover: 10% darker background for primary; accent color at 10% background for secondary
- Button active/pressed: brief 80ms opacity dip to 0.7

**Interaction:**
- "Copy to clipboard": copies plain-text version of the reconstructed log; button label changes to "Copied!" for 1500ms then reverts
- "Download .txt": triggers browser download of plain-text log as `voicefill-export.txt`
- "Try another file" link (below chat log, full-width center): returns to Upload Screen and resets state

### State 4 — "Without Media" Error Screen

Replaces the Upload Screen when a "without media" export is detected. Not a modal — the screen itself transforms (per D-05).

**Layout:**
- Same centered column as Upload Screen
- Error heading: "This export doesn't include audio files" — Heading size, normal weight, dark ink
- Explanation paragraph (Body size, dark ink, 1.5 line-height):
  "WhatsApp only includes voice messages when you export with media. To get the audio files, re-export the chat:"
- Numbered instruction list (Body size, dark ink, 1.5 line-height):
  1. "Open the chat in WhatsApp"
  2. "Tap the three-dot menu → More → Export Chat"
  3. "Choose 'Include Media' when prompted"
  4. "Share or save the resulting .zip file, then drop it here"
- "Try again" button below the list — same secondary button style as Results Screen actions
- Fine print below button: "The 'Without Media' option exports text only — voice messages are not included." — Body size, muted

**Interaction:**
- "Try again": returns to Upload Screen and resets all state

### State 5 — Input Mode: Parse-Only (.txt without audio)

Triggered when the user loads a raw `.txt` file (INPUT-04). No dedicated screen — the Results Screen renders with all voice lines annotated as `[Audio not available]`. Summary line reads: "0 of Y voice messages available — audio not included in this export."

This is a results-screen variant, not a distinct screen.

## Interaction Contracts

### Drag and Drop
- Drop zone responds to `dragenter`, `dragover`, `dragleave`, `drop` events
- Visual drag-enter state: accent background at 10% opacity + border opacity 100%
- Only `.zip` files accepted; non-ZIP drops silently ignored (drop zone returns to idle, no error shown)
- Both drag-and-drop and the file picker (`<input type="file" accept=".zip">`) trigger the same processing flow

### Animated Ellipsis (Processing Screen)
- CSS-only animation — no JavaScript `setInterval`
- Implementation: three `<span>` elements each containing `·`, animated with `@keyframes` opacity cycling at 600ms intervals, staggered by 200ms
- No spinner, no progress bar, no percentage — Phase 1 parsing is too fast for numeric progress to be meaningful

### Clipboard Copy
- Uses `navigator.clipboard.writeText()` — available in Electron renderer without special permissions
- On success: button label changes to "Copied!" for 1500ms, then reverts to "Copy to clipboard"
- On failure (clipboard API unavailable): fall back to `document.execCommand('copy')` on a hidden `<textarea>`; show "Copied!" on success, show "Copy failed — try selecting and copying manually" inline for 3000ms on failure

### File Download
- Creates an object URL from a `Blob` containing the plain-text content, triggers click on a hidden `<a download="voicefill-export.txt">` element, then revokes the URL

### Screen Transitions
- No CSS transitions between screens — instant swap via `display: none` / `display: block`
- Rationale: calm and deliberate aesthetic; transitions would add visual noise for a functional tool
- Exception: Processing Screen minimum display time of 300ms prevents jarring flash if parsing is instantaneous

## Component Inventory

All components are hand-authored HTML/CSS — no library dependencies.

| Component | HTML Element | Notes |
|-----------|-------------|-------|
| Drop zone | `<div>` with drag event listeners | Role: `region`, aria-label: "File upload area" |
| File picker | `<input type="file" accept=".zip">` | Visually hidden; triggered by "Browse for file" button |
| Browse button | `<button type="button">` | Triggers hidden file input via `.click()` |
| Processing indicator | `<p>` with three `<span>` children | CSS animated ellipsis |
| Sticky header | `<header>` | `position: sticky; top: 0` |
| Action button (primary) | `<button type="button">` | Copy to clipboard |
| Action button (secondary) | `<button type="button">` | Download .txt |
| Chat log | `<pre>` or `<div>` with monospace | Contains rendered message lines |
| Message line | `<div>` | Contains timestamp `<span>`, sender `<span>`, body `<span>` |
| Error screen | `<main>` variant | Same column layout as upload screen |
| Reset link | `<button type="button">` styled as link | "Try another file" |

Accessibility notes:
- All interactive elements reachable by keyboard (Tab order follows visual order)
- `aria-live="polite"` on processing status line for screen reader announcements
- `aria-label` on drop zone describing the accepted file type
- Color is never the sole differentiator — annotations use italic in addition to color
