    Add `parseChatText(rawText)` function (exported):
    - Strip UTF-8 BOM: `rawText.replace(/^﻿/, '')`
    - Strip Unicode directional marks: `.replace(/[‎‏‪-‮]/g, '')`
    - Regex `LINE_START`: `/^(\d{1,2}\/\d{1,2}\/\d{2,4},\s*\d{1,2}:\d{2}(?::\d{2})?(?:\s*[AP]M)?)\s*[-–]\s*(.+)/`
      (includes both ASCII hyphen and en-dash U+2013 in the separator character class)
    - Regex `VOICE_WITH_MEDIA`: `/^.+\.opus \(file attached\)$/`
    - Regex `VOICE_NO_MEDIA`: `/^<Media omitted>$/`
    - Parse logic exactly as in Research Pattern 2:
      - Iterate lines; for each line matching `LINE_START`, extract timestamp and rest
      - Split rest on first `: ` using `indexOf(': ')` and `slice` — NOT `split(': ')[0]` (Pitfall 3)
      - If no colon: `type: 'system'`
      - If content matches `VOICE_WITH_MEDIA`: `type: 'voice'`, extract `basename = content.replace(' (file attached)', '').trim()`
      - If content matches `VOICE_NO_MEDIA`: `type: 'voice-omitted'`
      - Otherwise: `type: 'text'`
      - Non-matching lines append to current message's content as continuation
    - Return `Message[]`

    Add `matchVoiceToAudio(messages, audioFiles)` function (exported):
    - Per Research Pattern 3: walk messages, for each `type: 'voice'`, look up `msg.basename` in `audioFiles` Map
    - If found: `msg.matched = true; msg.audioEntry = audioFiles.get(msg.basename);` (store ZipObject reference — do NOT read bytes in Phase 1)
    - If not found: `msg.annotation = '[Audio file missing]'` (ERR-03)
    - Track: `voiceTotal`, `voiceMatched`, `hasOmitted` (true if any voice-omitted found)
    - Return `{ messages, voiceTotal, voiceMatched, hasOmitted }`

    Add `detectExportMode({ audioFiles, hasOmitted, voiceMatched })` function (not exported, internal):
    - Per Research Pattern 4: if `hasOmitted && audioFiles.size === 0 && voiceMatched === 0`, return `'without-media'`
    - Otherwise return `'with-media'`
    - Critical: parse-only .txt mode (INPUT-04) has no audioFiles at all but voice lines reference basenames (not `<Media omitted>`) — mode will be `'with-media'` correctly per Pitfall 5

    Add `assemblePlainText(messages)` function (exported):
    - Per Research Pattern 5 and D-09, D-10:
      - Skip `type: 'system'` messages
      - `type: 'voice'` matched: body = `'[Voice message: transcription pending]'` (D-09)
      - `type: 'voice'` unmatched: body = `'[Audio file missing]'` (ERR-03 / D-10)
      - `type: 'voice-omitted'`: body = `'[Audio not available]'` (D-10 / INPUT-04)
      - `type: 'text'`: body = `msg.content`
      - Line format: `${msg.timestamp} - ${msg.sender}: ${body}`
    - Return joined string with `\n` separator

    Update `parseZip(file)` to use the full pipeline:
    - After existing `zip.forEach` and `chatEntry.async('string')` (already implemented in Plan 01-01):
    - Call `parseChatText(text)` to get `messages`
    - Call `matchVoiceToAudio(messages, audioFiles)` to get `{ messages, voiceTotal, voiceMatched, hasOmitted }`
    - Call `detectExportMode({ audioFiles, hasOmitted, voiceMatched })` to get `mode`
    - Call `assemblePlainText(messages)` to get `plainText`
    - Return `{ mode, messages, plainText, stats: { voiceTotal, voiceMatched }, rawLines: [] }` (rawLines no longer needed but kept for backward compat)

    Add `parseFolder(fileList)` function (exported, async):
    - INPUT-03: receives a `FileList` from `webkitdirectory` input
    - Iterate `fileList` entries: for each file, extract `file.name` as basename
    - Find `chatEntry` file where `name === '_chat.txt'` (or first `.txt` file as fallback)
    - Build `audioFiles` Map: `audioFiles.set(file.name, file)` for each `.opus` file (note: for folder mode, the entry is a `File` object, not a ZipObject — compatible for Phase 1 since we never read bytes; Phase 2 will adapt)
    - If no chatEntry: throw `new Error('No _chat.txt found in selected folder')`
    - Read chatEntry text: `const text = await chatEntry.text()`
    - Run same pipeline: `parseChatText`, `matchVoiceToAudio`, `detectExportMode`, `assemblePlainText`
    - Return same `{ mode, messages, plainText, stats }` shape

    Add `parseTxt(file)` function (exported, async):
    - INPUT-04: receives a single `.txt` `File` object (no audio)
    - Read text: `const text = await file.text()`
    - Call `parseChatText(text)` to get messages
    - No audio to match — create empty `audioFiles = new Map()`
    - Call `matchVoiceToAudio(messages, audioFiles)` — all voice lines will be unmatched
    - Call `detectExportMode({ audioFiles, hasOmitted, voiceMatched: 0 })` — mode will be `'with-media'` (voice lines reference filenames, not `<Media omitted>`) per Pitfall 5
    - Call `assemblePlainText(messages)` — unmatched voice lines become `[Audio file missing]`

The annotation `[Audio unreadable]` is a Phase 2 concern (when bytes are decoded). Phase 1 parser.js includes the `'[Audio unreadable]'` string as a comment referencing ERR-02 so Phase 2 knows where to inject it.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User chat text → parser | Sender names, message content, timestamps — all untrusted; can contain any characters |
| Folder FileList → basename Map | File names from OS directory listing; can contain path-like characters |
| .txt file content → parser | Raw text; may contain any Unicode; BOM and directional marks expected |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | XSS via sender names / message bodies in DOM | mitigate | All user content set via `textContent` exclusively; `innerHTML` forbidden for user data; enforced in both parser.js and ui.js; accepted by Plan 01-01's pattern; this plan adds no new DOM paths using innerHTML |
| T-02-02 | Tampering | Unicode directional marks causing sender/body misparse | mitigate | `parseChatText` strips `‎, ‏, ‪-‮` immediately on input, before any regex matching (PARSE-04) |
| T-02-03 | Spoofing | "Without media" mis-detection (parse-only .txt incorrectly routed to error screen) | mitigate | `detectExportMode` only triggers on `hasOmitted && audioFiles.size === 0 && voiceMatched === 0`; parse-only .txt has voice lines referencing filenames (not `<Media omitted>`) so `hasOmitted` is false; Pitfall 5 documented |
| T-02-04 | Denial of Service | Extremely long _chat.txt (millions of lines) causing UI hang | accept | Phase 1 parses synchronously on main thread; risk is low for realistic WhatsApp exports (max ~100K lines); Phase 2 can move parsing to Worker if needed; no mitigation in Phase 1 |
| T-02-05 | Information Disclosure | Error messages exposing internal paths or stack traces | mitigate | Error handler surfaces only `err.message` (user-friendly strings defined in parser.js); stack traces never reach DOM |
| T-02-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages installed in this plan; all code is hand-authored JS extending Plan 01-01's files |
</threat_model>
