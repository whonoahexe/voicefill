    Create `assets/js/parser.js` as an ES module:

    Export `parseZip(file)` — async function that takes a browser `File` object and returns `{ rawLines: string[], audioFiles: Map }`:
    - Validate input: if `file.name` does not end in `.zip` (case-insensitive), throw `new Error('Not a ZIP file')`
    - Add 500MB size guard: if `file.size > 500 * 1024 * 1024`, throw `new Error('ZIP file exceeds 500MB limit')`
    - Call `await JSZip.loadAsync(file)` — if it rejects, catch and rethrow as `new Error('Invalid or corrupt ZIP file')`
    - Call `zip.forEach((relativePath, zipEntry) => { ... })` to iterate all entries:
      - Skip entries where `zipEntry.dir === true`
      - Extract `basename = relativePath.split('/').pop()` (strips any subfolder prefix — per Pitfall 1)
      - If `basename === '_chat.txt'`: store as `chatEntry = zipEntry`
      - Else if `basename.endsWith('.txt') && chatEntry === null`: store as `chatEntry = zipEntry` (fallback)
      - Else if `basename.endsWith('.opus')`: `audioFiles.set(basename, zipEntry)`
    - If `chatEntry === null` after iteration: throw `new Error('No _chat.txt found in ZIP')`
    - Read chat text: `const text = await chatEntry.async('string')`
    - Split to lines: `const rawLines = text.split('\n').filter(l => l.trim().length > 0)`
    - Return `{ rawLines, audioFiles, rawText: text }`

    Note: In this Walking Skeleton plan, `parseZip` returns raw unprocessed lines. Full parsing (BOM stripping, voice detection, sender extraction) happens in Plan 01-02's `parseChatText` function which will be added to parser.js.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User filesystem → Browser memory | User-supplied File object; content is untrusted |
| ZIP archive → JS strings | JSZip parses binary ZIP; inner content (sender names, message text) is untrusted user data |
| JS strings → DOM | All user-data strings must reach DOM only via `textContent`, never `innerHTML` |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | ZIP path traversal (zip-slip) | mitigate | JSZip 3.8.0+ sanitizes `..` path components automatically; our code additionally uses basename-only Map — full paths never reach filesystem (in-memory only) |
| T-01-02 | Tampering | XSS via chat content in DOM | mitigate | All user-supplied content (sender names, message bodies, timestamps, filenames) set exclusively via `element.textContent`; `innerHTML` forbidden for any user data — enforced in parser.js and ui.js |
| T-01-03 | Denial of Service | Large ZIP causing browser OOM | mitigate | Reject files > 500MB before JSZip.loadAsync; audio ZipObject references stored lazily (bytes never read in Phase 1) |
| T-01-04 | Tampering | Malicious non-ZIP with .zip extension | mitigate | JSZip.loadAsync rejects non-ZIP binary; rejection caught and surfaced as user-friendly error; no filesystem write occurs |
| T-01-05 | Information Disclosure | Stack traces in UI error messages | mitigate | Error handler sets `textContent = err.message` (user-friendly string), never the full Error object or stack |
| T-01-SC | Tampering | npm/pip/cargo installs | mitigate | No npm/pip/cargo installs in this plan; JSZip is vendored via direct CDN URL download (jszip.min.js); Package Legitimacy Audit in RESEARCH.md confirms jszip@3.10.1 is approved |
</threat_model>
